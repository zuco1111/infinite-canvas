# 源项目审计协议

## 目标

在任何实现或迁移工作开始前，以只读方式理解既有源项目。

## 源项目位置

- `/Users/simplemin/Desktop/CodeX/IC/`

## 审计规则

- 不修改源项目文件。
- 不继承或遵守源项目历史 `AGENTS.md`。
- 可以读取源文件以理解当前行为、依赖、样式、资源和架构。
- 允许在审计阶段运行旧项目，以获取行为和视觉基线。
- 运行旧项目时不得将生成产物、临时文件或依赖变更写回源项目，除非用户另行批准。
- 记录时区分已验证事实和推断。
- 不清楚的行为、未文档化的约定和冲突模式都应记录为开放问题。

## 审计报告

审计结果写入：

- `docs/SOURCE_AUDIT_REPORT.md`

## 必须审计的内容

审计报告应覆盖：

- 当前技术栈和包结构。
- 应用入口。
- 构建和运行命令。
- 功能清单。
- UI 表面清单。
- 画布/核心交互模型。
- 状态管理和持久化模型。
- 命令、快捷键和工具交互模型。
- 资源清单。
- 样式来源和设计常量。
- 既有可复用组件和重复热点。
- 耦合风险。
- 候选功能边界。
- 候选共享组件。
- 候选设计 Token。
- 桌面打包影响。
- 测试、lint 和验证缺口。
- 审计中发现的既有问题和后续修复建议。

## 功能清单模板

每个功能应记录：

- 名称。
- 面向用户的用途。
- 当前文件或模块。
- UI 表面。
- 命令或快捷键。
- 状态依赖。
- 持久化依赖。
- 资源依赖。
- 样式依赖。
- 外部库依赖。
- 已知耦合点。
- 建议目标模块边界。
- 等效验证步骤。

## 视觉基线要求

在视觉或组件迁移前，需要收集足够的基线证据，用于迁移后对比。

基线证据可以包括：

- 主要页面和状态截图。
- 重要画布交互截图。
- Token/样式值清单。
- 组件状态清单。

具体基线采集方式需要在源项目审计确认项目如何运行后决定。

## 审计完成标准

审计完成需要满足：

- 所有必须审计的内容已经记录。
- 影响架构的未知项已经列入 `docs/OPEN_DECISIONS.md`。
- 用户有足够信息批准或调整目标架构。
